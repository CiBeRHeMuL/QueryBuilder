### 2. Step-by-step Data Flow

**Общий процесс документирования (на один файл):**

1. **Анализ файла:** Прочитать файл, понять его роль в архитектуре.
2. **Добавление `// region MODULE_CONTRACT`:** Вставить в начало файла (после `use`-блоков) блок с PHPDoc `@moduleContract`, `@purpose`, `@scope`, `@input`, `@output`, `@invariants`, `@modulemap`.
3. **Добавление `// GREP_SUMMARY:`:** Одна строка с ключевыми словами для grep-поиска.
4. **Добавление `// STRUCTURE:`:** Одна строка с мини-блок-диаграммой алгоритма (для функций/классов с логикой).
5. **Добавление `// region FUNC_...` / `// region CLASS_...` / `// region METHOD_...`:** Обернуть существующие функции/классы/методы в регион-теги с DOMAIN/CONCEPT/TECH ключевыми словами.
6. **Обогащение PHPDoc:** Добавить `@purpose`, `@io`, `@complexity` к существующим `@param`/`@return`.
7. **Создание `_module_contract.php`:** Для каждого неймспейса создать файл-контракт с `@moduleContract`, `@modulemap`, `@usecases`.
8. **Верификация:** `vendor/bin/php-cs-fixer check --config .php-cs-fixer.dist.php` (или `composer php-cs-check`) — файл не должен иметь ошибок. Единственный инструмент верификации стиля.

**Приоритизация (5 волн):**

| Волна | Модули | Файлов | Причина приоритета |
|-------|--------|--------|-------------------|
| 1 | Grammar (3), Builder (1), Helper (1), Exception (1), Expr (10) | ~16 | Ядро системы — наиболее критично для понимания |
| 2 | Query + Query/Interface + Query/Trait | ~18 | Fluent-интерфейс запросов — вторая по важности группа |
| 3 | Expr/\* (Cte, Group, Join, Lock, Order, SetOperation, Table, Update, Window) | ~18 | Выражения-композиты SQL |
| 4 | Enum/\* (все поднеймспейсы) | ~13 | Перечисления — простые файлы, быстро документируются |
| 5 | DB-specific (MySql/\*\*, PgSql/\*\*) | ~20 | Диалектные расширения — завершающая волна |

### 3. Acceptance Criteria

- [ ] **Criterion 1:** Все ~85 PHP-файлов содержат `// region MODULE_CONTRACT` с PHPDoc-блоком `@moduleContract`.
- [ ] **Criterion 2:** Все ~85 файлов содержат `// GREP_SUMMARY:` (ключевые слова через запятую).
- [ ] **Criterion 3:** Все классы обёрнуты в `// region CLASS_ClassName ... // endregion CLASS_ClassName`.
- [ ] **Criterion 4:** Все public-методы обёрнуты в `// region METHOD_methodName ... // endregion METHOD_methodName` с `@purpose`, `@io`.
- [ ] **Criterion 5:** Все функции/методы с алгоритмической логикой (более 2 строк) содержат `// STRUCTURE:` с мини-блок-диаграммой.
- [ ] **Criterion 6:** Созданы `_module_contract.php` для **каждого** неймспейса (35 файлов) с `@moduleContract`, `@modulemap`, `@usecases`.
- [ ] **Criterion 7:** `composer php-cs-check` (или `vendor/bin/php-cs-fixer check`) проходит без ошибок на всех изменённых файлах. Это единственный инструмент верификации стиля кода.
- [ ] **Criterion 8:** Ни одна строка бизнес-логики не изменена — только добавление комментариев/тегов.
- [ ] **Criterion 9:** Все region-теги сбалансированы (каждый `// region` имеет парный `// endregion`).

### 4. Риски и ограничения

| Риск | Меры |
|------|------|
| Большой объём (~85 файлов × 5-10 мин) — ~7-14 часов работы | Разбивка на 5 волн, каждая может выполняться независимым subagent |
| Код в активной разработке — документация может устареть | Документируем текущее состояние; `@changes LAST_CHANGE` фиксирует версию |
| Отсутствие тестов — нельзя автоматически верифицировать, что документация не сломала код | `composer php-cs-check` + визуальная проверка diff |
| Разные паттерны кода в разных модулях (enum, interface, trait, class) — нужны разные шаблоны документации | Определить шаблоны для каждого типа сущности до начала работы |

### 5. Шаблоны документации по типам сущностей

**Для простых DTO/Value Objects (Expr, BuiltQuery, Literal, DefaultValue):**
- `MODULE_CONTRACT` → краткий, указывает роль в AST
- `GREP_SUMMARY` → ключевые слова
- `CLASS_` → регион вокруг класса
- `METHOD_` → регионы вокруг методов
- Без `STRUCTURE:` (тривиальная логика)

**Для классов с алгоритмической логикой (AbstractGrammar, HExpr, ValueBuilder, Window):**
- Всё вышеперечисленное
- `STRUCTURE:` для ключевых методов
- `@complexity` в PHPDoc

**Для интерфейсов (все Query/Interface/\*, ExprInterface, GrammarInterface):**
- `MODULE_CONTRACT` → контрактные обязательства
- `GREP_SUMMARY`
- `INTERFACE_` → регион вокруг интерфейса
- `@purpose` в PHPDoc интерфейса

**Для трейтов (все Query/Trait/\*):**
- Аналогично классам, но `TRAIT_` вместо `CLASS_`

**Для enum (все Enum/\*):**
- `MODULE_CONTRACT` → смысл перечисления
- `GREP_SUMMARY`
- `ENUM_` → регион вокруг enum
- `METHOD_` → для `getSql()` и других методов

**Для `_module_contract.php`:**
- Только PHPDoc блок, без классов/функций
- `@moduleContract`, `@purpose`, `@scope`, `@modulemap`, `@usecases`
